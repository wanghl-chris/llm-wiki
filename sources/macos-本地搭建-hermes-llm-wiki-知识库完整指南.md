# 摘要
本文是基于真实部署经验整理的macOS本地Hermes LLM Wiki搭建完整指南，覆盖环境配置、功能操作、问题排查、最佳实践全流程，帮助用户快速搭建基于大模型的个人第二大脑知识库。系统支持多格式文档自动解析、全文检索、飞书集成等能力，通过16项交互式菜单实现低门槛操作，无需记忆复杂命令即可完成知识全生命周期管理。

# 核心观点
1. Hermes LLM Wiki是轻量化的本地个人知识库系统，依托大模型能力实现非结构化资料的自动结构化解析、索引与检索，无需复杂服务器部署即可在macOS端运行。
2. 系统采用「收件箱收集-结构化分层存储-版本控制」的架构设计，搭配16项覆盖全场景的交互式菜单，大幅降低个人知识管理的操作门槛。
3. 部署门槛极低，仅需Python 3.10+环境与火山引擎Ark API密钥，核心依赖轻量化，仅占用约500MB磁盘空间。
4. 遵循收件箱优先、统一标签治理、全量版本控制等最佳实践，可大幅提升知识库的运行效率与可维护性，后续可平滑扩展知识图谱、自然语言问答等进阶能力。

## 5. 常见问题与解决方案
### 问题1：Python字符串转义Bug
- 现象：运行时抛出`SyntaxError: unexpected character after line continuation character`错误
- 原因：字符串替换时错误将`\n`作为字面量写入代码，而非真实换行符
- 解决方案：调整字符串写法，将转义字符替换为实际换行：
  ```python
  # 错误写法
  content = content.replace('old', '\n# 调用 LLM')
  # 正确写法
  content = content.replace('old', '''
# 调用 LLM''')
  ```

### 问题2：菜单编号对齐问题
- 现象：新增菜单项后，显示编号与实际处理逻辑不匹配
- 原因：仅更新了菜单显示文本，未同步更新处理逻辑分支
- 解决方案：每次新增菜单项必须同步更新3处：`print_menu()`显示文本、对应处理函数、`main()`中的`elif choice == "N"`分支

### 问题3：生成字段代码泄露
- 现象：生成的index.md页面直接显示Python代码而非执行结果
- 原因：f-string语法错误，将代码逻辑写入字符串字面量而非执行后注入

## 6. 最佳实践
### 6.1 知识库组织原则
1. 收件箱机制：优先快速收集资料，后续集中整理，避免打断工作心流
2. 结构化输出：通过大模型统一生成「YAML元数据+结构化正文」的标准页面格式
3. 标签治理：维护统一的关键词体系，便于后续内容关联与分析
4. 版本控制：所有知识库内容纳入Git管理，支持变更追溯与版本恢复

### 6.2 交互设计原则
1. 飞书优先：所有操作支持飞书端完成，无需打开终端
2. 数字驱动：菜单采用数字选择模式，无需记忆命令行指令
3. 渐进披露：优先展示常用功能，高级功能按需展开
4. 容错设计：异常场景提供清晰提示，避免直接抛出堆栈信息

# 关键概念
1. **Hermes LLM Wiki**：运行在macOS本地的轻量化个人知识管理系统，依托大模型能力实现非结构化资料的自动结构化、索引与检索，定位为用户的个人第二大脑。
